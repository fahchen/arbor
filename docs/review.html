—a tree of stores,
spoken as JSON.

A page-scoped, server-authoritative runtime. One BEAM process owns a hierarchy of stateful stores, validates every render against a typed shape, and pushes RFC 6902 patches to a typed client. LiveView semantics where they fit; principled deviations recorded as decisions.

01What is Arbor

Phoenix LiveView taught us that a page can be a single, server-owned process. Arbor takes that lesson and reshapes it for typed, headless clients: instead of a tree of HTML, a tree of stores — each declaring the slice of JSON it owns — speaks to whatever rendering surface the client provides.

One BEAM process per connected page hosts a dynamic tree of stateful stores. Each store declares the public shape of its render output in a state block (state do … end). The runtime composes children explicitly through child(...) placeholders, validates every resolved render against the declared shape, computes the structural minimal RFC 6902 diff against the prior cycle, and ships the patch to the client over a Phoenix Channel.

The client is given two artefacts: the patch envelope, and a code-generated TypeScript type matching the store's state block. There is no HTML, no DOM, no opinion about what the client paints — only typed JSON moving in two directions over a single duplex channel.

Why typed JSON? The same page can drive a web SPA, a mobile native, or a headless test harness. The wire is the contract; the renderer is the application's choice.

Server · One process per page · Owns assigns & identity · Typed render output · Hooks for cross-cutting · No PubSub or persistence baked in.

Wire · Phoenix Channel ref-replies for command outcomes · Separate patch pushes carrying RFC 6902 ops + stream-ops · Versioned · No application-level resync.

Client · Receives initial state via a single replace at root path · Applies subsequent ops in order · Owns stream materialization · TypeScript types generated from the state block.

02Mental model

A page is a tree of stores. Each node owns a slice of the wire shape; children are placed explicitly by the parent's to_state/1; identity is the tuple (parent_path, module, id).

One map, two roles

A single socket.assigns map (LV-aligned, see Phoenix.Socket fields) holds parent-supplied values and store-internal state. The attr macro is a compile-time annotation: it enforces required parents at the parent's child(...) call site, and contributes to typespec and TypeScript codegen — but at runtime everything flows into socket.assigns.

The public shape — what the client sees — is a separate concern declared by the state block (state do … end). The to_state/1 function projects socket.assigns into that shape; nothing else reaches the wire. Whenever assigns change, the runtime re-runs to_state/1 and diffs the result against the previous output.

“The state block declares the public output shape. socket.assigns is the private workshop. The same contract drives validation, JSON Patch diffing, Elixir typespecs, and TypeScript codegen.” — Glossary, state block entry

Socket fields

Arbor's socket mirrors Phoenix.Socket's shape so handlers, hooks, and helpers all read the same struct. The fields a store author touches most:

FieldTypePurpose
assignsmap()The single state container. Holds parent-supplied values (declared via attr) and store-internal values (set in mount/1 and handlers). The only field to_state/1 reads from.
idString.t()The store node's id within its parent (the second component of the identity tuple). For the root page store, this is the page identifier; for children it's the value passed through child(Module, id: ..., ...).
parent_path[atom() \| String.t()]Ordered path from the root to this node's parent. Combined with module and id forms the runtime identity used for memoization and command routing.
modulemodule()The store module owning this node. Read-only.
endpoint · topic · transport_pidmodule() · String.t() · pid()Phoenix Channel scaffolding for this session. Read-only; provided so hooks and helpers can broadcast or push outside the standard envelope flow when needed.
privatemap()Reserved for runtime bookkeeping (hook table, async ref tracking, pending stream ops). Do not read or write directly; use the corresponding helpers (attach_hook/4, stream_insert/4, etc.).

Helpers like assign/2,3, update_assign/3, attach_hook/4, stream/4, assign_async/3,4, start_async/3,4, cancel_async/2,3, and stream_async/4 all take a socket as the first argument and return a new socket — chainable with |>.

03The DSL, in one breath

A complete store, exercising every primitive in the language. Read the comments inline; the rest of this page expands on each piece.

defmodule MyApp.Stores.ChatRoomStore do
  use Arbor.Store

  # Inputs from the parent (compile-time only; values flow into socket.assigns).
  attr :room_id,      String.t(), required: true
  attr :current_user, User.t(), required: true
  attr :on_leave,     (%{reason: atom()} -> any()), required: true

  # The state block — public output shape. Drives Elixir typespec, TS codegen,
  # validation, and the diff target. Stream slots are declared here too.
  state do
    field  :room_id,       String.t()
    field  :title,         String.t()
    field  :unread_count,  integer()
    field  :composer_text, String.t()
    field  :error,         String.t() | nil
    field  :status,        %{type: :live} | %{type: :paused, reason: String.t()}
    # item_key may be a 1-arity fn (item) or a 2-arity fn (item, socket).
    # Use the 2-arity form when the key depends on store state — e.g.,
    # &"msg-#{&2.assigns.room_id}-#{&1.id}".
    stream :messages, MessageState.t(), item_key: &"msg-#{&1.id}", limit: -200
  end

  # Client-callable commands. The payload block is optional —
  # a payload-less command is just `command :name`.
  command :ping

  command :send do
    payload :text, String.t()
  end

  command :leave do
    payload :reason, String.t()
  end

  def mount(socket) do
    Phoenix.PubSub.subscribe(MyApp.PubSub, "room:#{socket.assigns.room_id}")

    {:ok,
     socket
     |> assign(:title,         Rooms.title!(socket.assigns.room_id))
     |> assign(:composer_text, "")
     |> assign(:unread_count,  0)
     |> assign(:status,        %{type: :live})
     |> attach_hook(:audit,      :before_command, &MyApp.Hooks.Audit.log/4)
     |> attach_hook(:rate_limit, :before_command, &MyApp.Hooks.RateLimit.check/4)
     |> stream_async(:messages, fn -> {:ok, Chat.recent(socket.assigns.room_id, 50)} end)}
  end

  def handle_command(:send, %{text: ""}, socket),
    do: {:reply, %{ok: false, error: "empty"}, socket}

  def handle_command(:send, %{text: text}, socket) do
    {:ok, msg} = Chat.post(socket.assigns.room_id, socket.assigns.current_user, text)

    {:reply, %{ok: true, id: msg.id},
     socket
     |> assign(:composer_text, "")
     |> stream_insert(:messages, msg, at: 0, limit: -200)}
  end

  def handle_command(:leave, %{reason: r}, socket),
    do: {:noreply, invoke(socket, :on_leave, %{reason: String.to_atom(r)})}

  def handle_info({:message_received, msg}, socket),
    do: {:noreply,
          socket
          |> stream_insert(:messages, msg, at: 0, limit: -200)
          |> update_assign(:unread_count, &(&1 + 1))}

  # The state function — re-run after every assigns change; its
  # return value is diffed against the prior output to produce the patch.
  def state(socket) do
    %{
      room_id:       socket.assigns.room_id,
      title:         socket.assigns.title,
      unread_count:  socket.assigns.unread_count,
      composer_text: socket.assigns.composer_text,
      error:         socket.assigns.error,
      status:        socket.assigns.status,
      messages:      socket.assigns.messages
    }
  end
end

Anatomy

Macro / callRole
attrCompile-time declaration of a parent-supplied assign. Validates required at parent's child(...) call site; informs typespec and codegen. No runtime namespace — values land in socket.assigns.
state block · fieldThe public output shape. Drives the validation hook, the diff target, and both Elixir and TypeScript types.
commandClient-callable command and its payload schema. A built-in hook validates payloads on :before_command.
streamStream slot. Configures :item_key and :limit. The server retains only an ordered set of item keys after flush.
attach_hookThe sole extension primitive. Attach at any node, in any handler, on any of the five stages.
to_state(socket)Returns a value matching the state block. May contain child(...) placeholders the runtime resolves bottom-up.

04Commands & the hook pipeline

The command pipeline is one ordered list. There is no middleware macro; everything is attach_hook. Built-in hooks (schema validation, render-output validation) are attached by the runtime's mount path; authors may detach or replace them.

Callbacks vs. hooks

Two flavours of code participate in the lifecycle, with different signatures and authority.

Callbacks — store-defined functions on the module

CallbackSignatureReturns
mount/1(socket){:ok, socket}
update/2(new_assigns, socket){:ok, socket}
to_state/1(socket)state map matching the state block
handle_command/3(name, payload, socket){:noreply, socket} · {:reply, payload, socket}
handle_info/2(message, socket){:noreply, socket}
handle_async/3(name, result, socket){:noreply, socket}
terminate/2 root only(reason, socket)term

Children invoke parent-supplied callbacks by calling invoke(socket, :name, payload); the value at socket.assigns.name is just an inline closure the parent passed through child(...). There is no separate handle_callback dispatcher — pattern-match on the closure body itself.

Hooks — attached at runtime via attach_hook(socket, id, stage, fun)

StageSignatureReturns
:before_command(name, payload, socket){:cont, socket} · {:halt, socket} · {:halt, reply, socket}
:after_command(name, reply, socket){:cont, socket} · {:halt, socket}
:handle_async(name, result, socket){:cont, socket} · {:halt, socket}
:handle_info(message, socket){:cont, socket} · {:halt, socket}
:after_to_state(state_output, socket)socket (read-only stage; LV-aligned)

Hooks attached on a child store fire only for that node's events; cross-cutting concerns attach at the root and pattern-match on the path. Each store keeps its own hook table. There is no mount-stage hook; setup goes inside mount/1 directly.

def mount(socket) do
  socket
  |> attach_hook(:audit,      :before_command, &Audit.log/3)
  |> attach_hook(:rate_limit, :before_command, &RateLimit.check/3)
  |> attach_hook(:trace,      :after_to_state,    &Trace.snapshot/2)
  |> then(&{:ok, &1})
end

Errors crash; the client reconnects

Arbor mirrors LiveView's let-it-crash posture for malformed or impossible commands. Unknown paths, undeclared commands, schema-failing payloads, and any other "the runtime cannot service this request" condition raise inside the runtime; the page process exits; the supervisor restarts a fresh runtime; the transport drops; the client reconnects and replays its initial state from the new mount. There is no wire enum of error categories — the runtime never produces a structured error reply for these cases.

Authorization and graceful denials live in hook halts, not crashes. A :before_command hook may return {:halt, %{ok: false, reason: "unauthorized"}, socket}; the transport reply carries that payload with channel status :ok, and the client decides how to surface it. The handler is skipped; no patch ships.

“If LiveView would crash, Arbor crashes. If the application has a graceful answer, it returns it as a hook reply. The wire stays narrow: :ok with a payload, or the connection drops.” — Mirrors BDR-0003 let-it-crash semantics

05Streams: the server forgets

For long, append-mostly collections — a chat scrollback, an activity feed — Arbor mirrors LiveView's stream API exactly. The wire delivers per-item ops; the server retains only an ordered set of item keys after each flush.

The full LiveView API surface

The function shapes mirror Phoenix.LiveView's stream API verbatim, swapping dom_id for item_key. See LiveView's stream/4 docs for prior art.

# Seed or reset a stream slot. Items are added in reverse order; reset: true clears first.
stream(socket :: t(), name :: atom() | String.t(), items :: Enumerable.t(), opts :: keyword() \\ []) :: t()

# Configure :item_key fn and defaults; must run before any stream/4 call for the same name.
stream_configure(socket :: t(), name :: atom() | String.t(), opts :: keyword()) :: t()

# Upsert by computed item_key. Opts: :at, :limit, :update_only.
stream_insert(socket :: t(), name :: atom() | String.t(), item :: any(), opts :: keyword() \\ []) :: t()

# Delete the item by its computed item_key.
stream_delete(socket :: t(), name :: atom() | String.t(), item :: any()) :: t()

# Delete by a precomputed item_key, when the item itself isn't on hand.
stream_delete_by_item_key(socket :: t(), name :: atom() | String.t(), item_key :: String.t()) :: t()

To refresh a stream's contents, call stream(socket, name, fresh_items, reset: true) directly with items already in hand — or, if you need a loading flash while an async fetch runs, use stream_async(socket, name, fun, reset: true). There is no dedicated reload primitive (see BDR-0022).

06Async & AsyncResult

Background work flows through three primitives borrowed from LiveView, plus one Arbor extension. Every async value on the wire is an AsyncResult — a three-field struct keyed on a status enum (:loading | :ok | :failed); pattern-match on the status to drive the UI.

# Variant 1 — assign_async writes the AsyncResult to socket.assigns automatically.
socket |> assign_async(:profile, fn -> {:ok, %{profile: Users.fetch!(id)}} end)

# Variant 2 — start_async routes the result to handle_async/3; assigns are author-managed.
socket |> start_async(:warm_cache, fn -> Cache.warm(id) end, timeout: 5_000)

# Variant 3 — stream_async composes both: AsyncResult drives the loading flag,
# items seed the stream slot.
socket |> stream_async(:messages, fn -> {:ok, Chat.recent(room_id, 50)} end)

“Tasks are linked to the runtime; they live and die with it. A child node disappearing does not actively cancel its tasks — results that arrive for a vanished node are silently discarded as :lazy_discard.” — BDR-0019, accepted

Arbor diverges from LiveView in one notable place: a raise inside handle_async/3 is caught and recorded as [:arbor, :async, :exception]; the runtime continues. The reasoning is documented in BDR-0020. A bug in async result-processing should not flap the entire page on every completion.

07The wire, on display

Three envelopes move between server and client. Reading them in order is enough to predict any moment of the runtime.

Command (client → server)

{
  "type": "command",
  "path": ["filters"],
  "command": "change_query",
  "payload": { "query": "shirt" }
}

Reply (server → client, ref-correlated)

// Successful command, no payload from the handler.
{ "status": "ok", "payload": {} }

// Successful command, handler returned a payload via {:reply, payload, socket}.
{ "status": "ok", "payload": { "order_id": "ord_42" } }

// Auth (or any other) hook halt — graceful denial. status stays "ok"; the
// payload's ok flag tells the client.
{ "status": "ok", "payload": { "ok": false, "reason": "unauthorized" } }

// Malformed command (unknown path / undeclared command / schema fail) does
// NOT produce a structured reply — the runtime crashes and the channel drops.
// The client reconnects and re-mounts.

Patch push (server → client)

{
  "type": "patch",
  "base_version": 41,
  "version": 42,
  "ops": [
    { "op": "replace", "path": "/selected_product_id", "value": "prod_123" }
  ],
  "stream_ops": [
    { "op": "insert", "path": ["chat"], "stream": "messages",
      "item_key": "msg-7", "at": 0, "limit": -100,
      "data": { "id": 7, "body": "hi" } }
  ]
}

The first envelope after a fresh mount is always a single replace at path "", carrying the entire root output. The version counter resets to zero on every reconnect — recovery is the reconnect path; there is no application-level resync command.

08Twenty decisions, one shape

Each named tension was resolved by a Behaviour Decision Record. The numbering is sparse because two records were pruned during review — BDR-0016 (stream-server-forgets-values, intrinsic to the stream concept) and BDR-0021 (the :timeout async option, an uncontested addition). Reading the rest is a faithful tour of the design's load-bearing choices.

09Deliberate non-goals

What Arbor will not do is as load-bearing as what it does. Each entry is a closed door with a known reason on the other side.

  • One process per child store. Mailbox ordering and merge complexity outweigh isolation gains for an MVP.
  • CRDT or offline-first sync. Out of scope; this is a server-authoritative runtime.
  • Event sourcing. The runtime persists nothing; applications layer their own.
  • Templated UI composition. No HEEx, no JSX. The wire is JSON; layout is the client's job.
  • Slot composition. Children are placed by child(...); slots are unnecessary.
  • Built-in PubSub. Use Phoenix.PubSub.subscribe/2 directly and handle_info/2.
  • Built-in persistence. An application pattern, not a runtime primitive.
  • Application-level resync command. Recovery is the reconnect path.
  • Subtree-replace fallback. Always emit the structural minimal diff.
  • Per-child unmount/terminate callback. Mirrors Phoenix.LiveComponent.
  • Server-side stream value retention. Stream values live only on the client.
  • Async-result auto-persistence. No persist: :ok_only; the application implements via hooks.
  • JSON Patch move, copy, test ops. Out of scope for MVP.
